# Redis

---

## 1. Redis 为什么快

**答:** 基于内存 + 单线程减少锁竞争和上下文切换 + 高效数据结构和 IO 模型。

- 最根本是内存，单线程只是减少额外开销

---

## 2. 数据类型与场景

| 类型 | 特点 | 典型场景 |
|------|------|----------|
| String | 最基础，可存字符串/数字 | 缓存、计数器、分布式锁标记、Session |
| Hash | 字段集合 | 用户信息、商品详情、配置项 |
| List | 有序可重复，双端操作 | 消息队列、最新动态列表 |
| Set | 无序不重复 | 去重、共同好友、标签 |
| ZSet | 有序集合，带 score | 排行榜、热搜、延迟任务 |

---

## 3. RDB vs AOF

| 对比 | RDB | AOF |
|------|-----|-----|
| 原理 | 定时快照 | 追加写命令日志 |
| 优点 | 文件小、恢复快 | 数据更安全、丢失少 |
| 缺点 | 可能丢最近数据 | 文件大、恢复慢 |

一句话：RDB 记结果，AOF 记过程。

---

## 4. 过期删除 vs 内存淘汰

- **过期删除:** 针对已到期 key，惰性删除（访问时清）+ 定期删除（后台抽查清）
- **内存淘汰:** 内存满了还要写入时触发，常见策略：
  - `noeviction` 报错
  - `allkeys-lru` 所有 key 淘汰最近最少使用
  - `volatile-lru` 只在有过期时间的 key 中淘汰
  - `volatile-ttl` 优先淘汰快过期的

---

## 5. 缓存穿透 / 击穿 / 雪崩

| 问题 | 本质 | 解决方案 |
|------|------|----------|
| 穿透 | 查不存在的数据，每次打 DB | 缓存空值、布隆过滤器、参数校验 |
| 击穿 | 热点 key 失效，瞬时压力 | 互斥锁、逻辑过期、预热 |
| 雪崩 | 大量 key 同时失效 / Redis 挂了 | 过期时间加随机值、限流降级、Redis 高可用 |

---

## 6. BigKey / HotKey

- **BigKey:** value 太大或集合元素太多 → 传输慢、删除阻塞、内存不均
- **HotKey:** 访问频率极高 → 某节点压力大，失效时引发击穿
  - 解决：本地缓存、多副本分散、热点 key 不立即过期、限流

---

## 7. 缓存一致性

**答:** 先更新数据库，再删除缓存（Cache Aside）。

- 为什么不先删缓存：中间有读请求会把旧数据加载回来
- 为什么不更新缓存：缓存按需重建更简单，主动更新引入新失败点
- 双删策略（了解）：更新 DB 后删缓存，延迟再删一次，降低旧数据回填风险

---

## 8. 分布式锁

**答:** `SET key value NX EX seconds`，NX 保证互斥，EX 防死锁。

- value 存唯一标识（UUID），解锁前校验是自己的锁
- 解锁用 Lua 脚本：「判断 value + 删除 key」必须原子
- 为什么不能拆开加锁和设过期：中间宕机 → 死锁
- 为什么不能直接 DEL：可能误删别人的锁

### 风险

- 业务执行超过锁超时 → 锁提前过期（续期机制）
- Redis 主从切换 → 锁可靠性风险

### 适用场景

- 防重复下单/提交、分布式定时任务互斥、库存扣减

## 高频对比速记

| 对比 | 核心区别 |
|------|----------|
| RDB vs AOF | 快照 vs 命令日志 |
| 过期删除 vs 内存淘汰 | key 到期清理 vs 内存满了清理 |
| 穿透 vs 击穿 vs 雪崩 | 不存在 / 热点失效 / 大面积失效 |
| BigKey vs HotKey | 数据太大 vs 访问太多 |
