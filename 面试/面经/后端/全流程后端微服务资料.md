# 后端微服务架构速查

---

## 1. Servlet to Spring MVC

**核心概念**: Front Controller模式，DispatcherServlet统一处理所有HTTP请求，Controller只写业务逻辑

**关键点**:
- 原生Servlet需要每个类自己处理HTTP协议细节（参数解析、响应封装）
- Spring MVC的DispatcherServlet接管协议处理，Controller用`@RestController`声明式开发
- Spring底层仍运行在Servlet容器（Tomcat）上
- Web层职责：参数校验、协议转换、响应包装，**禁止写业务逻辑**

**易错点**: 不要在Service层引入HttpServletRequest，会导致业务层和Web层强耦合

## 2. IoC & DI

**核心概念**: 控制反转，对象由框架创建和管理，通过构造器注入依赖

**关键点**:
- 面向接口编程，依赖声明为`private final Interface`
- 推荐`@RequiredArgsConstructor`（Lombok）做构造器注入
- **禁止字段注入**（@Autowired字段），脱离容器无法单测
- 循环依赖会导致启动失败（Spring Boot 2.6+默认禁止）

**面试答法**: IoC把对象创建权交给框架，开发者只声明依赖接口，运行时由容器注入实现类，实现解耦和可测试性。

---

## 3. AOP

**核心概念**: 通过动态代理在方法执行前后插入横切逻辑（日志、事务、权限）

**关键点**:
- 底层是JDK动态代理或CGLIB代理
- 核心注解：`@Aspect`, `@Around`, `@Before`, `@After`
- 典型应用：`@Transactional`、`@Cacheable`、`@Async`
- **AOP失效场景**：同类内部方法调用时，this非代理对象，AOP不生效

**易错点**: 不要在同一个类里调用本类的`@Transactional`或`@Async`方法，代理会失效

---

## 4. ThreadLocal

**核心概念**: 线程独享变量，避免参数透传（Parameter Drilling）

**关键点**:
- 典型应用：Spring Security的`SecurityContextHolder`
- 存储用户上下文（userId、租户ID）跨层级传递
- **必须在请求结束时`ThreadLocal.remove()`**，否则线程池复用会导致数据泄露
- 适用场景：单线程执行链路的全局状态传递

**面试答法**: Tomcat使用线程池，线程复用导致不清理ThreadLocal会串号，必须在finally或afterCompletion里清理。

---

## 5. 声明式事务 @Transactional

**核心概念**: 通过AOP代理，捕获异常自动回滚，方法执行完自动提交

**关键点**:
- 推荐写法：`@Transactional(rollbackFor = Exception.class)`
- 默认只回滚RuntimeException和Error，不回滚Checked Exception
- **Try-Catch陷阱**：catch住异常不抛出，事务不会回滚
- 底层依赖ThreadLocal绑定同一个数据库Connection

**易错点**:
1. 异常被吞掉不会回滚，需在catch块里`throw new RuntimeException(e)`或调用`TransactionAspectSupport.currentTransactionStatus().setRollbackOnly()`
2. 同类内部调用无效（AOP失效）

## 6. 消息队列 MQ

**核心概念**: 异步解耦，Fire-and-Forget，削峰填谷

**关键点**:
- 典型场景：发邮件、生成PDF、延迟任务（30分钟未支付自动取消）
- 核心组件：Producer发消息、Consumer消费消息、Queue缓冲
- **幂等性**：At-least-once投递，需防重（唯一索引/Redis Token）
- **手动ACK**：业务成功后再确认，防止消费失败消息丢失
- 死信队列（DLQ）：处理永远失败的消息

**面试答法**: MQ解决同步阻塞导致的性能瓶颈，通过异步解耦提升吞吐量，代价是最终一致性。

## 7. Redis缓存

**核心概念**: Cache Aside模式，读先查缓存，未命中再查DB并写回缓存

**缓存三大问题**:
| 问题 | 原因 | 解法 |
|------|------|------|
| 缓存击穿 | 热点Key过期瞬间，高并发打穿DB | 分布式锁（Mutex），只允许一个线程查DB |
| 缓存穿透 | 恶意请求不存在的Key，绕过缓存直击DB | 布隆过滤器 / 缓存空值（短TTL） |
| 缓存雪崩 | 大量Key同时过期，DB瞬时压力暴增 | TTL加随机抖动（如2h±10min） |

**关键点**:
- 设置合理TTL，防止冷数据占用内存
- 更新策略：先更新DB，再删除缓存（Cache Invalidation）
- 实际开发用Spring Cache（`@Cacheable`, `@CacheEvict`）简化代码

## 8. 服务发现与RPC

**核心概念**: 微服务通过注册中心（Nacos/Eureka）动态寻址，Feign声明式RPC调用

**关键点**:
- 服务启动时向Nacos注册IP和端口，心跳保活
- `@FeignClient(name="inventory-service")`：像调本地方法一样调远程服务
- Feign自动负载均衡、序列化、HTTP请求
- **必须配置超时**：connectTimeout和readTimeout，默认超时太长
- **熔断器（Circuit Breaker）**：连续失败N次后熔断，执行Fallback降级逻辑

**面试答法**: Nacos解决"你在哪"，Feign解决"怎么调"，熔断器防止雪崩效应。

## 9. API Gateway

**核心概念**: 微服务统一入口，承担鉴权、路由、限流职责

**关键点**:
- 前端只访问一个域名（如api.example.com），网关路由到内网服务
- 核心功能：①统一鉴权（JWT验证） ②动态路由（`/order/**` → order-service） ③限流
- **底层是Netty + Reactive**：非阻塞NIO，Event Loop模型，少量线程扛高并发
- **禁止Blocking I/O**：不要在Gateway Filter里用同步JDBC或`Thread.sleep()`

**配置示例**:
```yaml
spring.cloud.gateway.routes:
  - id: order-service
    uri: lb://order-service  # lb = LoadBalancer
    predicates:
      - Path=/api/order/**
    filters:
      - StripPrefix=1
```

## 10. 分布式事务

**核心概念**: 跨数据库操作保证一致性，强一致性用Seata，最终一致性用MQ

**两种方案**:

### Seata AT模式（强一致性）
- `@GlobalTransactional`开启全局事务
- TC（事务协调者）统一协调多个服务的提交/回滚
- 底层记录Undo Log实现补偿
- **代价**：全局锁，性能差，不适合高并发场景

### MQ最终一致性（推荐）
- 订单服务提交本地事务后发MQ消息
- 库存服务消费消息扣库存，失败则重试或发补偿消息
- 用时间换空间，允许短暂不一致
- 高并发场景首选（如秒杀、抢购）

**面试答法**: 强一致性牺牲性能，最终一致性牺牲实时性。根据业务选型：核心资金用Seata，高并发用MQ。

## 11. Docker容器化

**核心概念**: 镜像打包应用+依赖+运行环境，消除"本地能跑线上不行"问题

**Dockerfile示例**:
```dockerfile
FROM eclipse-temurin:17-jre-alpine
VOLUME /tmp
COPY target/app.jar app.jar
ENTRYPOINT ["java", "-jar", "/app.jar"]
```

**关键点**:
- 镜像分层：基础镜像（OS+JDK）+ 依赖 + 业务代码
- **Volume挂载**：持久化数据（如MySQL数据目录）必须挂载到宿主机
- docker-compose：编排多服务（MySQL+Redis+App）一键启动
- K8s：容器编排，自动扩缩容、自愈

**易错点**: 不要把MySQL数据库直接跑在容器里不挂Volume，容器删除数据就没了

## 12. CI/CD

**核心概念**: 代码提交后自动测试、打包、部署，实现快速交付

**GitHub Actions流程**:
```yaml
on:
  push:
    branches: [main]
jobs:
  deploy:
    steps:
      - uses: actions/checkout@v3
      - uses: actions/setup-java@v3
      - run: mvn clean package
      - uses: appleboy/ssh-action@master
        with:
          host: ${{ secrets.SERVER_IP }}
          script: |
            cd /opt/app
            docker-compose down
            docker-compose up -d --build
```

**蓝绿发布**:
- Blue（线上老版本） + Green（新版本）
- 新版本部署到Green，测试通过后切流量，秒级回滚

**关键点**:
- 必须有自动化测试，否则变成"自动部署Bug机器"
- **禁止硬编码密码**：用Repository Secrets存储敏感信息
- 部署策略：蓝绿发布（零停机）/ 灰度发布（逐步放量）

## 架构全景图

```
[用户] → [API Gateway]
         ↓
    [订单服务] ←→ [Nacos注册中心]
         ↓ Feign RPC
    [库存服务]
         ↓
    [Redis缓存] + [MySQL] + [RabbitMQ]
         ↓
    [Docker容器] + [CI/CD流水线]
```

**核心哲学**:
1. 单一职责：Controller/Service/Mapper各司其职
2. 面向接口编程：依赖抽象不依赖具体
3. 异步解耦：MQ削峰填谷
4. 防御性编程：熔断、降级、限流
5. 基础设施即代码：Docker + CI/CD自动化
