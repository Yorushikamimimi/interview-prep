# 后端知识概念速查

---

## 性能指标

- **RT (Response Time):** 系统从接收请求到返回结果的总时间
- **P99:** 99% 请求的响应时间上限。大厂排查性能看 P99/P99.9，不看平均值（被正常请求掩盖）

---

## 高并发六大问题

| 问题 | 本质 |
|------|------|
| 超卖 Overselling | 并发控制/锁失效，多线程同时读旧库存执行 UPDATE |
| 热点竞争 Hotspot | 海量线程争抢同一行锁/同一把 Redis 锁，CPU 消耗在上下文切换 |
| DB 瓶颈 | 慢 SQL / 缺索引 / 连接池打满，达到 IOPS 或连接数极限 |
| 重试风暴 Retry Storm | 下游超时 → 自动重试 → 流量放大数倍 → 链路压垮 |
| 读写放大 | 存储引擎底层机制，写 10B 实际磁盘重写整个 16KB 数据页 |
| 惊群效应 Thundering Herd | 热点缓存失效/锁释放时，瞬间唤醒所有等待线程 |

## 后端分层架构

| 层 | 职责 | 规范 |
|----|------|------|
| Controller | 参数校验、协议转换、统一响应包装 | 禁止写业务逻辑（瘦 Controller） |
| Service | 业务编排、事务控制、核心逻辑 | @Transactional、调度多个 Mapper |
| Mapper/DAO | 纯 CRUD，屏蔽存储差异 | 不写 if-else 业务判断 |
| Entity | 数据库表的 Java 映射（POJO） | 不写业务逻辑 |

## ABA 问题与版本号解决

- **ABA 问题:** CAS 只比较值（1==1），但值可能经历了 A→B→A 变化
- **解决:** 加 version 字段（只增不减），CAS 比较 version 而非业务值
- SQL: `UPDATE stock SET amount=amount-1, version=version+1 WHERE id=? AND version=?`

---

## MySQL 索引相关

- **回表:** 二级索引只存主键 ID，需要回主键索引查完整数据（两次查询）
- **覆盖索引:** 查询字段全在索引上，无需回表，性能大幅提升
- **GIN 倒排索引:** 适合 JSONB/数组/全文检索，值→行 ID 映射（"从下往上找"）

---

## MySQL 锁

- **行锁:** WHERE 命中主键/唯一索引，只锁一行，并发高
- **表锁:** WHERE 未命中索引，锁整张表，并发降为 0（P0 事故）
- **死锁:** 两个事务互相持有对方需要的锁。规避：固定顺序加锁（先锁 ID 小的）

## 其他概念

- **Tomcat:** Web 服务器/Servlet 容器，Spring Boot 内嵌（从 1.x 就有，不是 3.x 才有）
- **Gateway:** Nginx / Spring Cloud Gateway，负责流量分发、鉴权、限流
- **防抖 Debounce:** 用户停止操作一段时间后才发请求，降低无效 QPS
- **Redis SETNX 防重:** `SETNX key "processing" EX 10`，key 存在则拒绝（必须加过期时间防死锁）
- **JSONB:** 写入时预解析，读取免解析，比 TEXT/JSON 快成百上千倍
