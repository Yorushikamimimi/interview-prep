# 任务调度系统

> 一句话：基于 PostgreSQL SKIP LOCKED 的 Pull 型异步任务调度系统，核心是数据库驱动的任务编排而非媒体平台。

---

## 简版（1 分钟）

这个项目我定义为一个基于 PostgreSQL 的 Pull 型异步任务调度系统，媒体下载只是示例场景，核心是任务调度。架构上做了控制面和执行面分离：Java Spring Boot 作为 Control Plane 负责任务创建、状态管理、持久化和 SSE 推送；Python Worker 作为 Data Plane 通过 HTTP 轮询拉取任务、执行后回写结果。

核心链路：前端创建任务 → 后端写入 sys_media_task 表（PENDING）→ Worker 调用 /pending 接口 → 后端事务内 SELECT ... FOR UPDATE SKIP LOCKED 选未锁任务 → 更新 RUNNING → Worker 执行 → 成功回写 SUCCESS / 失败进重试。还做了 RUNNING 超时回收防止僵尸任务。

最有价值的三点：用数据库行锁而非 MQ 做并发认领完成最小闭环；落地了状态机、失败重试和超时回收；控制面与执行面分离使边界清晰。

---

## 完整版（2-3 分钟）

这个项目我定义为一个基于 PostgreSQL 的 Pull 型异步任务调度系统。整体架构做了控制面和执行面分离：Java Spring Boot 作为 Control Plane，Python Worker 作为 Data Plane，Worker 不直连数据库，只通过 HTTP API 工作。

**并发认领（核心）：** 没有引入 MQ，直接把 PostgreSQL 当任务队列。Worker 轮询 /pending 接口时，后端在事务内用 SELECT ... FOR UPDATE SKIP LOCKED 锁住一条任务再更新为 RUNNING，多 Worker 并发时自动跳过已锁任务。关键点：选任务和改状态必须在同一事务内，否则仍可能重复消费。

**状态机与可靠性：** 任务状态收敛为 PENDING → RUNNING → SUCCESS/FAILED。失败不立即重试，按 retry_count/max_retry/next_retry_at 做指数退避。定时扫描长时间停留 RUNNING 的任务做超时回收，防 Worker 崩溃后任务卡死。

**控制面/执行面分离：** Worker 不直连 DB，事务、鉴权、状态规则收敛在 Java 控制面。好处是边界清晰、可跨语言扩展；代价是多一层 HTTP 往返。

**trade-off：** 用 DB 代替 MQ，换来更低复杂度和更快落地，牺牲了扩展性。语义更接近"至少一次"，真正幂等靠业务保证。

**诚实说不足：** 鉴权只是 Bearer Token；Worker 在线状态是进程内心跳非注册中心；下载文件依赖 Worker 本机路径，多机部署不稳定；无完整单测和 CI。

## 高频追问

**Q: 为什么不用 MQ？**
A: 目标是最小可交付闭环，SKIP LOCKED 能直接基于数据库行锁实现多 Worker 并发认领，少一个中间件，部署维护成本更低。缺点是扩展性不如 MQ。

**Q: 为什么不用 Redis 分布式锁？**
A: 核心问题是"安全领取一条任务记录"，任务本身在 DB 里，用 DB 事务+行锁更自然，少一层组件。

**Q: SKIP LOCKED 怎么避免重复消费？**
A: 关键是事务边界——同一事务内先 SELECT FOR UPDATE SKIP LOCKED 锁住任务，再立刻改状态为 RUNNING。查和改分开就可能重复。

**Q: Worker 为什么不直连数据库？**
A: 任务状态规则、事务控制、鉴权集中在 Java 控制面，Worker 只负责执行，边界清晰，也方便跨语言扩展。

**Q: 重试为什么用 next_retry_at？**
A: 防止失败后立即重试形成重试风暴，配合指数退避平滑瞬时故障。

**Q: 为什么要 RUNNING 超时回收？**
A: Worker 可能崩溃/断网/卡死，没有回收逻辑任务会永久卡在 RUNNING。

**Q: "至少一次"还是"恰好一次"？**
A: 至少一次。有失败重试和超时回收，任务可能重复执行，严格恰好一次要靠业务幂等保证。

**Q: 最大 trade-off？**
A: 用 DB 代替 MQ，更低复杂度+更快落地，但牺牲扩展性和解耦能力。

**Q: 继续演进先改什么？**
A: ①文件产物迁共享存储 ②Worker 状态升级为持久化注册 ③任务量增大则引入 MQ 解耦。
