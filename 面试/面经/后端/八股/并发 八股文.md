# 并发

---

## 1. 进程 vs 线程

**答:** 进程是资源分配单位（独立内存），线程是 CPU 调度单位（共享进程资源）。线程更轻量但共享资源易出并发问题。

---

## 2. 创建线程方式

- 继承 Thread
- 实现 Runnable（无返回值，不能抛受检异常）
- 实现 Callable + Future（有返回值，可抛异常）
- 实际开发交给线程池管理

---

## 3. sleep() vs wait()

| 对比 | sleep() | wait() |
|------|---------|--------|
| 所属 | Thread | Object |
| 是否释放锁 | 不释放 | 释放 |
| 使用要求 | 任意位置 | 必须在同步块内 |
| 唤醒方式 | 超时自动 | notify()/notifyAll() |

---

## 4. synchronized

**答:** Java 最基础的同步手段，本质是对象锁，保证原子性 + 可见性。

- 修饰实例方法 → 锁 this
- 修饰静态方法 → 锁 Class
- 修饰代码块 → 锁指定对象

---

## 5. volatile

**答:** 保证可见性 + 禁止部分指令重排序，**不保证原子性**。

- 适合：状态标记、开关变量、双重检查单例
- 不适合：i++ 这种读-改-写操作

---

## 6. CAS

**答:** Compare And Swap，比较当前值与预期值，相同则更新，否则失败重试。无锁并发思想，AtomicInteger 底层依赖它。

- 问题：ABA 问题、自旋开销、只能保证单变量原子性

---

## 7. 乐观锁 vs 悲观锁

| 对比 | 悲观锁 | 乐观锁 |
|------|--------|--------|
| 思路 | 先加锁再操作 | 先操作再校验 |
| 实现 | synchronized / Lock | CAS / 版本号 |
| 适用 | 冲突频繁 | 冲突不高 |

---

## 8. synchronized vs Lock

| 对比 | synchronized | Lock (ReentrantLock) |
|------|-------------|---------------------|
| 层面 | 关键字，语法级 | API 接口 |
| 释放锁 | 自动 | 手动 unlock |
| 灵活性 | 基础 | 可中断、可超时、可尝试(tryLock) |

简单同步优先 synchronized，需要更强控制用 Lock。

---

## 9. 线程池

### 为什么用

复用线程 + 控制并发数 + 统一管理任务。

### 核心参数（ThreadPoolExecutor）

| 参数 | 含义 |
|------|------|
| corePoolSize | 核心线程数 |
| maximumPoolSize | 最大线程数 |
| keepAliveTime | 非核心线程空闲存活时间 |
| workQueue | 任务队列 |
| handler | 拒绝策略 |

### 执行流程

**先核心 → 再队列 → 再非核心 → 最后拒绝**

### 拒绝策略

| 策略 | 行为 |
|------|------|
| AbortPolicy | 直接抛异常（默认） |
| CallerRunsPolicy | 提交方线程自己执行（反压） |
| DiscardPolicy | 直接丢弃 |
| DiscardOldestPolicy | 丢弃队列最旧任务 |

### 为什么不用 Executors

默认配置不可控：FixedThreadPool 无界队列可能 OOM，CachedThreadPool 线程数可能失控。

### execute() vs submit()

- execute() 无返回值
- submit() 返回 Future，可获取结果和异常

## 10. 死锁

**答:** 多线程互相等待对方释放资源，谁都走不下去。

### 四个必要条件

互斥、请求并持有、不可剥夺、循环等待

### 避免

- 固定加锁顺序
- 减少锁嵌套
- tryLock() + 超时

### 排查

jstack 导出线程栈，查看阻塞线程和锁等待关系。

---

## 11. ThreadLocal

**答:** 给每个线程一份独立变量副本，用空间换线程隔离（不是加锁）。

- 场景：用户上下文、TraceId、数据库连接
- 风险：线程池中不及时 remove() → 内存泄漏、数据污染

---

## 12. AtomicInteger

**答:** 基于 CAS 的原子类，线程安全地做自增/自减/更新，比加锁更轻量。

- volatile 只保证可见性，AtomicInteger 保证原子更新
