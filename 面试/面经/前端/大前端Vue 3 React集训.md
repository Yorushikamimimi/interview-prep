# Vue 3 / React 前端速查

---

## 1. 核心概念

### State（状态）
**定义**: 会变化且变化后影响页面显示的数据

**示例**:
- 是否已点赞 `liked: boolean`
- 点赞数 `likeCount: number`
- 按钮是否在请求中 `isLiking: boolean`

**公式**: `State = 驱动UI变化的数据`

---

### Props（属性）
**定义**: 从父组件传给子组件的数据，子组件只读不可修改

**生动比喻**:
- 父组件像老师，子组件像学生
- 老师发给学生一张试卷（props），学生只能看和做，不能修改试卷内容
- 如果学生想改分数，必须让老师（父组件）改

---

### Re-render（重新渲染）
**定义**: State变化后，框架重新计算组件输出并更新DOM

**触发条件**:
- 组件自身state变化
- 父组件传入的props变化
- 订阅的全局状态变化

---

## 2. 虚拟DOM

### 定义
虚拟DOM是用JS对象描述真实DOM结构的数据表示，框架通过Diff算法计算最小更新范围。

### 核心流程
**State变化 → 生成新虚拟DOM → Diff算法对比 → 计算最小变更 → 批量更新真实DOM**

**比喻**: 像改Word文档，不是每打一个字就保存一次，而是打完一段话后统一保存（批量更新）

### 为什么需要虚拟DOM
| 问题 | 虚拟DOM解法 |
|------|-------------|
| 直接操作DOM慢 | 先在内存中计算，批量更新一次 |
| 不知道更新哪里 | Diff算法精确计算变化范围 |
| 频繁更新卡顿 | 合并多次变更，一次性渲染 |

### 典型场景：点赞按钮
**操作**: 点击点赞 → `liked` 从 `false` 变 `true`

**虚拟DOM工作**:
1. 对比按钮颜色属性 `color: gray → red`
2. 对比点赞数 `1234 → 1235`
3. 只更新这两个DOM属性，不重建整个页面

## 3. 状态管理

### 本地State vs 全局State

| 类型 | 定义 | 适用场景 | 示例 |
|------|------|----------|------|
| **本地State** | 组件内部管理，只影响当前组件 | 弹窗开关、输入框内容、按钮loading | `isModalOpen`, `inputValue` |
| **全局State** | 跨组件共享，多处使用 | 用户信息、购物车、主题模式 | `currentUser`, `cartItems` |

### 状态管理库
- **Pinia** (Vue 3推荐)
- **Redux** (React经典方案)
- **Zustand** (React轻量方案)

### 点赞场景状态拆分

| 数据 | 归属 | 原因 |
|------|------|------|
| `liked` (已点赞) | 全局 | 列表页和详情页都要显示 |
| `likeCount` (点赞数) | 全局 | 多处展示，需同步 |
| `isLiking` (请求中) | 本地 | 只影响当前按钮交互 |

## 4. 高频交互优化（点赞/收藏场景）

### 乐观更新（Optimistic UI）
**定义**: 前端先展示成功状态，请求失败再回滚

**流程**:
```
用户点击 → 立刻变红+1 → 发请求 → 成功✓ / 失败→回滚灰色-1
```

**适用场景**: 低风险操作（点赞、收藏），失败率低且可重试

---

### 防抖 vs 节流 vs 按钮置灰

| 方案 | 定义 | 适用场景 | 点赞是否适用 |
|------|------|----------|--------------|
| **Debounce（防抖）** | 停止操作N秒后才执行 | 搜索框输入（停止输入500ms后搜索） | ❌ 首次点击必须立刻响应 |
| **Throttle（节流）** | N秒内最多执行1次 | 滚动加载、窗口resize | ❌ 不是主方案（作为辅助） |
| **Pending置灰** | 请求中禁用按钮 | 提交表单、点赞、支付 | ✅ **主方案** |

### 点赞标准思路
**流程**: 点击 → 立刻高亮 → 置灰500ms → 发请求 → 成功保持/失败回滚

**比喻**: 像点餐
1. 点击"加一份鸡腿"按钮 → 立刻在购物车显示+1（让你有反馈）
2. 按钮变灰（防止你狂点）
3. 后台确认库存（发请求）
4. 成功→订单确认；失败→提示"鸡腿卖光了"并从购物车移除

## 5. API设计（点赞场景）

### 方案一：独立接口
```
POST   /songs/:id/like      # 点赞
DELETE /songs/:id/like      # 取消点赞
```

**优点**: 语义清晰，符合RESTful
**缺点**: 前端需判断当前状态决定调哪个接口

---

### 方案二：Toggle接口（推荐）
```
POST /songs/:id/like/toggle
```

**优点**: 前端无需判断状态，后端根据当前状态自动切换
**幂等保证**: `userId + songId` 唯一约束，重复请求返回相同结果

---

### 接口设计要点
1. **幂等性**: 同一请求多次执行结果一致
2. **去重**: 数据库唯一索引（userId + songId）
3. **状态回传**: 返回最新的 `liked` 状态和 `likeCount`
4. **错误处理**: 失败返回明确错误码和提示

## 6. SSR vs CSR

### 核心区别
| 维度 | SSR（服务端渲染） | CSR（客户端渲染） |
|------|-------------------|-------------------|
| **首屏HTML** | 服务端生成完整HTML | 返回空壳HTML + JS |
| **SEO** | 好，爬虫直接读到内容 | 差，需额外方案（预渲染） |
| **首屏速度** | 快，直接展示内容 | 慢，需等JS下载执行 |
| **服务器压力** | 高，每次请求渲染HTML | 低，只返回静态文件 |
| **交互响应** | 慢，需等JS hydration | 快，纯前端交互 |
| **开发复杂度** | 高，需考虑服务端环境 | 低，纯前端开发 |

---

### 业务选型

| 场景 | 推荐方案 | 原因 |
|------|----------|------|
| **营销落地页** | SSR | SEO优先，首屏快 |
| **后台管理系统** | CSR | 无需SEO，交互复杂 |
| **电商商品详情** | SSR + CSR混合 | 首屏SSR，交互CSR |
| **社交Feed流** | CSR + 骨架屏 | 个性化内容多，SEO次要 |

---

### 混合方案（推荐）
**核心思路**: 首屏SSR，交互CSR

**实现**:
1. 服务端渲染页面框架和首屏数据
2. 客户端接管后，点赞/评论等交互走CSR
3. 避免SSR渲染个性化内容（如"已点赞"状态）

**示例**: 歌曲详情页
- **SSR渲染**: 歌名、歌手、封面、基础点赞数（SEO需要，首屏快）
- **CSR渲染**: 当前用户是否已点赞、点赞交互、评论列表（个性化内容）

## 7. Vue 3 vs React 核心差异

### 响应式原理
| 框架 | 原理 | 代码风格 |
|------|------|----------|
| **Vue 3** | Proxy劫持，自动追踪依赖 | 声明式，修改data自动更新 |
| **React** | 显式调用setState触发更新 | 函数式，调用setState才更新 |

### 核心区别比喻

**Vue 3**: 像智能家居，你说"开灯"（修改data），灯自动亮（自动更新UI）
- 写法：`liked.value = true` → 按钮自动变红
- 特点：声明式，框架自动追踪依赖

**React**: 像遥控器，必须按遥控器按钮（调用setState）才能开灯
- 写法：`setLiked(true)` → 告诉React更新UI
- 特点：函数式，显式触发更新

## 8. 完整交互链路（点赞示例）

### 用户视角
**点击按钮 → 立刻变红 → 数字+1 → (500ms后可再次点击)**

### 技术链路
1. 用户点击
2. 前端乐观更新（liked=true, likeCount+1）
3. 按钮置灰500ms（防止重复点击）
4. 发送请求到后端
5. 成功→保持状态；失败→回滚+提示

**比喻**: 像网购下单
- 点击"立即购买" → 页面立刻显示"订单提交中"（乐观更新）
- 按钮变灰防止重复提交
- 后台确认库存和支付
- 成功→显示"订单成功"；失败→恢复按钮+提示"库存不足"

### 状态流转过程
**State变化 → 虚拟DOM计算差异 → 只更新变化的DOM → 用户看到界面变化**

**比喻**: 像修改PPT
- 改了一页的标题颜色（State变化）
- 软件计算只有这个标题需要重绘（Diff算法）
- 只刷新标题区域，不重绘整个页面（最小化更新）

## 9. 常见错误

### ❌ 错误1：Debounce用在点赞
**错误做法**: 用防抖，点击后等500ms才响应

**为什么错**: 用户点一下按钮，等半天没反应，以为没点上，继续狂点

**正确做法**: 立刻高亮按钮（给反馈）+ 置灰500ms（防重复）

---

### ❌ 错误2：只做前端防重，忽略服务端幂等
**错误做法**: 只在前端禁用按钮500ms

**为什么错**:
- 用户点击瞬间，按钮置灰前已经发出了请求
- 网络慢，第一个请求还没返回，用户刷新页面又点了一次
- 前端防不住所有情况

**正确做法**: 前端置灰（基础防御）+ 后端唯一索引（终极保障）
- **比喻**: 像银行转账，手机App上防止重复点击只是第一道防线，银行后台还要检查"是否已经转过"

---

### ❌ 错误3：全局状态滥用
**错误做法**: 把按钮的loading状态放到全局store

**为什么错**:
- 按钮loading只影响这一个按钮，不需要其他组件知道
- 如果放全局，页面上10个点赞按钮会相互影响

**正确做法**:
- **交互状态**（loading、弹窗开关）→ 本地state
- **数据状态**（点赞数、用户信息）→ 全局store
- **比喻**: 你的手机电量显示（本地），不需要告诉全班同学（全局）

---

### ❌ 错误4：视觉反馈=真实成功
**错误做法**: 点击后立刻弹Toast提示"点赞成功"

**为什么错**:
- 请求还没发完，可能会失败
- 用户看到"成功"但实际失败了，体验很差

**正确做法**:
- 按钮立刻变红（乐观更新UI）
- 不弹"成功"提示（静默成功）
- 只在失败时弹Toast提示错误
- **比喻**: 像发微信，发送中显示"发送中"，成功了不提示，失败了才显示红色感叹号

## 11. 面试答法模板

### 题目：设计高并发点赞功能

**核心思路**:
采用**乐观更新 + 按钮置灰 + 服务端幂等**的方案。前端点击后立刻高亮按钮并+1计数，同时置灰500ms防止连续点击。发送POST /like/toggle幂等接口，成功后更新全局状态，失败则回滚UI并Toast提示。

**技术要点**:
1. **状态拆分**: `liked`和`likeCount`放全局（多处展示），`isLiking`放本地（仅影响按钮交互）
2. **虚拟DOM**: State变化自动触发最小化DOM更新，只改按钮颜色和数字
3. **防重方案**: 前端按钮置灰 + 后端userId+songId唯一索引
4. **异常处理**: 请求失败回滚UI，下次用户可重试

**ROI权衡**:
点赞是低风险操作，允许短暂视觉不一致（乐观更新），换取即时反馈和后端压力降低。失败率低且用户可重试，不需要引入死信队列等重量级补偿机制。
