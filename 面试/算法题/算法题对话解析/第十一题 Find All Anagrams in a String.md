**

题目：Find All Anagrams in a String

给你两个字符串 s 和 p，找到 s 中所有是 p 的 anagram（异位词） 的子串起始下标。

示例：

- 输入：s = "cbaebabacd", p = "abc"
    
- 输出：[0, 6]
    
- 解释：
    

- s[0..2] = "cba"，是 "abc" 的异位词
    
- s[6..8] = "bac"，也是 "abc" 的异位词
    

---

## 先给结论

这题大概属于：滑动窗口（Sliding Window） + 计数统计（frequency counting） 题型。  
本质是：在字符串 s 里维护一个固定长度为 len(p) 的窗口，判断这个窗口里的字符组成是否和 p 一样。

---

## 已知事实

1. 题目本质  
    不是找完全相同的子串，而是找“字符种类和每种字符出现次数都一样”的子串。
    
2. 核心思路候选  
    最自然的方向是：
    

- 先统计 p 里每个字符出现次数
    
- 再在 s 上维护一个长度固定的窗口
    
- 比较窗口统计和 p 的统计是否一致
    

4. 为什么它像滑动窗口题  
    因为我们不是要枚举所有子串，而是只关心长度等于 p.length 的连续片段。  
    这种“固定窗口长度”的信号非常强，通常就会想到 Sliding Window（滑动窗口：让一个区间在数组/字符串上向右滑动，而不是每次重新算）。
    
5. 可能会用到的数据结构
    

- hash map：哈希表，记录字符 -> 次数
    
- 或 array[26]：如果题目默认只有小写字母，可以直接用长度 26 的数组统计，更快更简单
    

---

## 合理推测

1. 这题大概率有两个层次的解法：
    

- baseline solution：每个窗口都重新统计一次，再比较
    
- optimal solution：窗口右移时只增减两个字符的计数，不重复统计
    

3. 这题真正想练的不是字符串技巧，而是：
    

- 看到“固定长度子串”
    
- 联想到“固定窗口”
    
- 再把“异位词判断”转成“频次数组是否一致”
    

5. 面试里这题常常不是只要你会做，而是要你能讲清楚：
    

- 为什么窗口长度固定为 len(p)
    
- 为什么比较频次就够了
    
- 为什么优化后能从“重复统计”变成“增量更新”

## 核心术语一句话解释

- Anagram：异位词，指字符完全相同，但顺序可以不同。
    
- Sliding Window：滑动窗口，在序列上维护一个连续区间，随着右指针移动动态更新。
    
- Frequency Count：频次统计，记录每个字符出现了几次。

更准确的面试表达应该改成：

第一个窗口需要单独初始化，因为后续的滑动更新是建立在“已有窗口频次”的基础上的。只有先统计出第一个窗口的频次，后面窗口右移时才能通过减去左边字符、加上右边字符来增量更新，而不必重新统计整个窗口。

题目：Find All Anagrams in a String

示例：

- 输入：s = "cbaebabacd", p = "abc"
    
- 输出：[0, 6]
    

---

## 先给结论

这题的最优解是：固定长度滑动窗口 + 频次数组。  
核心做法是：

- 先统计 p 的字符频次
    
- 再维护 s 中一个长度为 p.length() 的窗口频次
    
- 窗口每次右移一格时，只做两件事：
    

- 左边出去的字符 -1
    
- 右边进来的字符 +1
    

- 每次判断两个频次数组是否一致，一致就记录当前起点
    

这是这题最标准、面试最常见的解法。

---

## 已知事实

### 1. 题型

这题属于：Sliding Window（滑动窗口）

一句话解释：

- 因为我们只关心长度固定为 p.length() 的连续子串，所以天然适合用固定窗口来扫一遍 s。
    

### 2. 本质

题目不是找和 p 完全相同的子串，而是找：

- 长度相同
    
- 字符频次相同
    

也就是 anagram（异位词）。

### 3. 为什么不用排序

也可以对每个窗口排序后比较，但那样太慢。  
更高效的方法是直接比较字符出现次数。

### 4. 核心数据结构

如果题目字符集是小写字母：

- 用 int[26] 最合适
    

因为：

- cnt[c - 'a'] 就能表示某个字符的出现次数
    
- 比 HashMap 更快、更稳
    

---

## 合理推测

### baseline solution（朴素解）

对 s 的每个长度为 k = p.length() 的子串：

- 重新统计一次频次
    
- 再和 p 比较
    

这个能做出来，但慢。

### optimal solution（最优解）

不要每次重算整个子串频次，而是：

- 先初始化第一个窗口
    
- 后面窗口右移时增量更新
    

这就是优化点。

## 需要验证

这题在 LeetCode 默认前提下通常是：

- s 和 p 由小写英文字母组成
    
- 返回所有合法起始下标
    
- 下标从 0 开始
    

如果字符集变大，比如有大小写、Unicode，通常要把 int[26] 换成 HashMap<Character, Integer>。

# 一、baseline 到 optimal 的思路

## 方案 A：baseline

### Core Idea

枚举所有长度为 k 的子串，每次重新统计频次。

### 为什么慢

因为相邻窗口高度重叠。  
比如：

- "cba"
    
- "bae"
    

其实只差两个字符：

- 'c' 出去
    
- 'e' 进来
    

没必要把整个窗口重新数一遍。

---

## 方案 B：optimal

### Core Idea

维护一个固定长度窗口的频次数组。

流程：

1. 统计 p 的频次
    
2. 统计 s 的第一个窗口频次
    
3. 如果相等，加入 0
    
4. 从第二个窗口开始滑动：
    

- window[s[i - k]]--
    
- window[s[i]]++
    

6. 每次判断 window 是否与 target 相等
    

---

# 二、最优解伪代码

k = p.length

if s.length < k:

    return []

  

target[26] = p 的频次

window[26] = s 前 k 个字符的频次

  

if window == target:

    ans.add(0)

  

for i from k to s.length - 1:

    移出字符 = s[i - k]

    移入字符 = s[i]

  

    window[移出字符]--

    window[移入字符]++

  

    if window == target:

        ans.add(i - k + 1)

  

return ans

# 三、Java 最终答案

import java.util.ArrayList;

import java.util.Arrays;

import java.util.List;

  

class Solution {

    public List<Integer> findAnagrams(String s, String p) {

        List<Integer> ans = new ArrayList<>();

  

        int n = s.length();

        int k = p.length();

  

        if (n < k) {

            return ans;

        }

  

        int[] target = new int[26];

        int[] window = new int[26];

  

        // 统计 p 的频次

        for (char c : p.toCharArray()) {

            target[c - 'a']++;

        }

  

        // 初始化第一个窗口

        for (int i = 0; i < k; i++) {

            window[s.charAt(i) - 'a']++;

        }

  

        if (Arrays.equals(window, target)) {

            ans.add(0);

        }

  

        // 滑动窗口

        for (int i = k; i < n; i++) {

            window[s.charAt(i - k) - 'a']--; // 移出左边字符

            window[s.charAt(i) - 'a']++;     // 加入右边字符

  

            if (Arrays.equals(window, target)) {

                ans.add(i - k + 1);

            }

        }

  

        return ans;

    }

}

# 四、复杂度分析

设：

- n = s.length()
    
- k = p.length()
    

## baseline

如果每个窗口都重新统计长度为 k 的频次：

- 窗口数量：n - k + 1
    
- 每个窗口统计一次：O(k)
    

所以：

- 时间复杂度：O((n - k + 1) * k)，通常写成 O(nk)
    
- 空间复杂度：O(1)  
    （如果字符集固定为 26）
    

## optimal

- 初始化频次：O(k)
    
- 窗口滑动 n - k 次
    
- 每次比较两个长度 26 的数组，代价是常数
    

所以：

- 时间复杂度：O(n)
    
- 空间复杂度：O(1)

# 五、容易错的点

## 1. 忘记先初始化第一个窗口

原因：

- 后续滑动更新必须基于一个已存在的窗口频次
    

## 2. 起始下标算错

当右指针在 i 时：

- 当前窗口起点是 i - k + 1
    

## 3. 边界条件漏掉

如果：

s.length() < p.length()

  

直接返回空数组。

## 4. 把“集合相同”误认为“异位词”

异位词判断看的是：

- 每个字符的频次  
    不是只看字符种类。
    

---

# 六、面试可复述版本

这题可以用固定长度滑动窗口解决。因为我们要找的是 s 中所有长度等于 p.length() 的异位词子串，所以窗口长度固定为 p.length()。  
先统计 p 的字符频次，再统计 s 的第一个窗口频次。如果相等，就把下标 0 加入答案。  
后面让窗口每次向右滑动一格：把左边移出的字符频次减一，把右边新进入的字符频次加一。每次更新后判断当前窗口频次是否和 p 的频次一致；如果一致，就记录当前窗口起始下标。  
因为字符集固定为 26 个小写字母，所以数组比较是常数级，整体时间复杂度是 O(n)，空间复杂度是 O(1)。

# 七、如果题目条件变化，怎么改

## 变化 1：字符不再只包含小写字母

把：

- int[26]
    

改成：

- HashMap<Character, Integer>
    

代价：

- 代码更复杂
    
- 常数更大
    

## 变化 2：不是找所有位置，而是只判断是否存在

那就：

- 一旦发现匹配，直接返回 true
    

## 变化 3：允许通配符或更复杂匹配

那就不再是简单频次统计题了，可能要引入：

- 双指针
    
- 哈希
    
- 更复杂状态维护

- 这题属于固定长度滑动窗口，本质是把异位词判断转成窗口频次和目标频次是否一致。
    

# 九、一句话核心 insight

- 这题最关键的突破点是：相邻窗口只差一个移出字符和一个移入字符，所以没必要每次重新统计整个窗口频次。
