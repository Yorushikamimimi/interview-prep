## 2. 项目防御要点（门诊系统）

### 狙击点1：CAS锁 vs Redis（50 QPS场景）

**面试官发难**: "才50 QPS，有必要搞这么复杂的锁吗？Redis不是大厂标配吗？"

**反杀答法**:
"正因为真实边界只有50 QPS，盲目引入Redis才是架构灾难！
- **技术选型**: MySQL CAS乐观锁（`WHERE current_stock >= quantity`）
- **ROI权衡**: 实现0超卖 + 降低运维成本 + 避免Redis脑裂风险
- **核心理念**: 用最便宜的架构解决当前最核心的问题"

---

### 狙击点2：JSONB + GIN倒排索引

**面试官发难**: "关系型数据库里用JSONB和GIN索引，这一般是搜索引擎干的事？"

**反杀答法**:
"门诊患者动态标签（过敏史、特殊体质）字段极度不固定：
- **传统方案痛点**: EAV模型→JOIN爆炸；加N个NULL字段→空间浪费
- **PostgreSQL方案**: JSONB存储（Schema-free）+ GIN倒排索引
- **性能提升**: 多维度全表扫描从秒级 → 50ms
- **定位**: 单体架构下替代沉重ES集群的最优解"

---

### 狙击点3：Dify + MQ防OOM

**面试官发难**: "大模型推理慢很正常，为什么要在网关层加MQ？"

**反杀答法**:
"单卡弱机环境下，显存是绝对物理瓶颈：
- **不加MQ后果**: 并发请求打满Tomcat线程池 → 全盘OOM
- **MQ作用**: 网关层削峰填谷，牺牲时间换空间
- **核心逻辑**: 宁可让用户排队，也必须保住核心推理服务不宕机"

## 4. 高并发OOM问题

### Root Cause分析

**菜鸟回答**: "访问量太大，对象太多，内存撑爆了"（❌废话）

**满分回答**:
"高并发本身不直接导致OOM，**根本原因是'高并发 + 下游阻塞'**：
- 慢SQL或外部接口超时 → 线程hang住
- Tomcat线程池耗尽 → 请求堆积
- 堆内存被线程栈和pending对象占满 → OOM"

---

### 生产抢救SOP

**❌ 错误做法**: 赶紧重启服务器（流量还在，3秒后再次OOM）

**✅ 正确流程**:
1. **保护现场**: JVM参数 `-XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/data/logs/` 自动生成.hprof快照
2. **紧急摘流**: Nginx/网关层摘除宕机节点，隔离后重启
3. **验尸分析**: MAT/Arthas打开.hprof文件，定位哪个类实例占90%内存，追踪到具体Controller接口

---

### 非技术妥协方案

**Product Engineer视角**:
"优秀的PE不会死磕代码。预估流量超系统极限时，主动要求产品：
- 排队等待动画（用户可接受）
- 展示本地缓存降级数据
- **前端按钮点击后置灰5秒** → 在客户端挡住80%重复并发"

## 5. 秒杀场景设计（50万并发抢1000份盲盒）

### 系统瓶颈分析

**最先宕机环节**:
1. **Tomcat线程池**: 默认200线程，50万并发瞬间耗尽
2. **数据库连接池**: 默认10-20连接，扛不住高并发写
3. **网络带宽**: 单机1Gbps带宽，50万请求直接打满

---

### 大厂规范方案（Redis Lua + MQ）

#### 前端防御
- 活动前按钮置灰（防提前刷接口）
- 点击后立刻置灰5秒 + 显示"排队中"
- 客户端挡住80%重复点击

#### 后端架构流程
**形象比喻**: 演唱会抢票系统
1. **Nginx限流**: 像演唱会入口，每秒只放10万人进场（其余直接提示"人太多，稍后再试"）
2. **Redis预扣减**: 大厅里有个电子屏显示剩余票数，抢到的人立刻减1，抢空立刻显示"售罄"（秒级响应）
3. **RabbitMQ排队**: 抢到票的1000人拿到"取票凭证"，去后台窗口慢慢办理（削峰）
4. **MySQL落单**: 后台工作人员从容地给这1000人打印正式门票（持久化存储）

#### 核心机制
**Redis Lua原子扣减**: 就像电子屏的"查库存→扣减"是一个原子动作，不会被打断。如果中间能插队，就会出现"10个人同时看到剩1张票，结果都抢到了"的超卖。

#### 为什么不直接MySQL
- **Redis内存操作**: 像从口袋掏东西（10万QPS轻松扛）
- **MySQL磁盘IO**: 像从保险箱取东西（5000 QPS就接近极限）
- **秒杀场景**: 50万人同时抢，直接怼MySQL就像50万人挤到一个保险箱门口，直接踩踏事故

## 7. 技术栈对比

### PostgreSQL vs MySQL

| 特性 | PostgreSQL | MySQL |
|------|------------|-------|
| **JSONB** | 原生支持，写入预解析 | JSON类型性能差 |
| **GIN倒排索引** | 内置，适合全文检索 | 需借助外部工具 |
| **MVCC** | 多版本并发控制，读写不阻塞 | InnoDB支持但略逊 |
| **复杂查询** | 强大，支持窗口函数/CTE | 相对弱 |
| **适用场景** | 数据分析、复杂查询、动态Schema | OLTP、简单查询、高并发写 |

### 选型建议
- **门诊动态标签场景**: PostgreSQL（JSONB + GIN）
- **普通电商订单**: MySQL（成熟生态）
- **库存扣减**: 两者都可（CAS机制相同）

## 9. 面试万能公式

### 技术选型公式
**当前业务量级 + 资源约束 + 运维成本 → 最简可用方案**

**示例**: 门诊50 QPS + 单机4GB内存 + 无专职运维 → MySQL CAS锁（而非Redis集群）

### 问题定位三步法
**表象 → Root Cause → 解决方案**

**示例**: OOM宕机 → 高并发+慢SQL导致线程池耗尽 → 限流+降级+优化SQL

### 答题结构四要素
1. **业务背景**: 门诊系统患者动态标签字段不固定
2. **技术选型**: PostgreSQL JSONB + GIN倒排索引
3. **ROI权衡**: 单体架构下不引入ES，降低运维成本
4. **结果数据**: 多维检索从秒级优化到50ms
