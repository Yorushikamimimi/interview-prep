# 任务调度系统

> 一句话：基于 PostgreSQL SKIP LOCKED 的 Pull 型异步任务调度系统，核心是数据库驱动的任务编排而非媒体平台。

---

## 简版（1 分钟）

这个项目我定义为一个基于 PostgreSQL 的 Pull 型异步任务调度系统，媒体下载只是示例场景，核心是任务调度。架构上做了控制面和执行面分离：Java Spring Boot 作为 Control Plane 负责任务创建、状态管理、持久化和 SSE 推送；Python Worker 作为 Data Plane 通过 HTTP 轮询拉取任务、执行后回写结果。

核心链路：前端创建任务 → 后端写入 sys_media_task 表（PENDING）→ Worker 调用 /pending 接口 → 后端事务内 SELECT ... FOR UPDATE SKIP LOCKED 选未锁任务 → 更新 RUNNING → Worker 执行 → 成功回写 SUCCESS / 失败进重试。还做了 RUNNING 超时回收防止僵尸任务。

最有价值的三点：用数据库行锁而非 MQ 做并发认领完成最小闭环；落地了状态机、失败重试和超时回收；控制面与执行面分离使边界清晰。

---

## 为什么不用 MQ？

目标是最小可交付闭环，SKIP LOCKED 能直接基于数据库行锁实现多 Worker 并发认领，少一个中间件，部署维护成本更低。缺点是扩展性不如 MQ。

## 为什么不用 Redis 分布式锁？

核心问题是"安全领取一条任务记录"，任务本身在 DB 里，用 DB 事务+行锁更自然，少一层组件。

## SKIP LOCKED 怎么避免重复消费？

关键是事务边界——同一事务内先 SELECT FOR UPDATE SKIP LOCKED 锁住任务，再立刻改状态为 RUNNING。查和改分开就可能重复。

## Worker 为什么不直连数据库？

任务状态规则、事务控制、鉴权集中在 Java 控制面，Worker 只负责执行，边界清晰，也方便跨语言扩展。

## 重试为什么用 next_retry_at？

防止失败后立即重试形成重试风暴，配合指数退避平滑瞬时故障。

## 为什么要 RUNNING 超时回收？

Worker 可能崩溃/断网/卡死，没有回收逻辑任务会永久卡在 RUNNING。

## "至少一次"还是"恰好一次"？

至少一次。有失败重试和超时回收，任务可能重复执行，严格恰好一次要靠业务幂等保证。

## 最大 trade-off？（任务调度系统）

用 DB 代替 MQ，更低复杂度+更快落地，但牺牲扩展性和解耦能力。

## 继续演进先改什么？（任务调度系统）

①文件产物迁共享存储 ②Worker 状态升级为持久化注册 ③任务量增大则引入 MQ 解耦。
