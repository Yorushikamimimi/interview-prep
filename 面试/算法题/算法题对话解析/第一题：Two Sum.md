# 一、为什么是 Time Complexity = O(n²)？

## 已知事实

假设数组里有 n 个元素。

Brute Force 的做法是：

- 第一个元素，去和后面所有元素配对
    
- 第二个元素，再去和后面所有元素配对
    
- 第三个元素，继续……
    

本质是在检查所有可能的二元组合 pair。

如果写成次数，大概是：

(n-1) + (n-2) + (n-3) + ... + 1

  

这个和的量级是：

n(n-1)/2

  

忽略常数和低阶项后，就是：

O(n²)

  

## 直觉理解

这不是“循环两层所以硬背 O(n²)”；  
而是因为你真的在做：

“每个元素，都要跟后面一批元素再扫一遍。”

这就是典型的 nested scan（嵌套扫描）。

---

# 二、为什么是 Space Complexity = O(1)？

## 已知事实

空间复杂度看的是：

额外申请的内存，是否会随着输入规模 n 变大而变大。

在 Brute Force 里，你通常只会用到这些东西：

- i
    
- j
    
- 可能再加一个临时和 sum
    

这些变量数量是固定的，不会因为数组从 10 个元素变成 10 万个元素，就多出一大堆新空间。

所以额外空间是常数级：

O(1)

  

## 你刚才容易混淆的点

你把“会做很多次计算”误以为“要占很多空间”。

这是两件不同的事：

- Time Complexity：做了多少计算
    
- Space Complexity：额外占了多少内存
    

大厂黑话里可以理解成：

- time 看的是 CPU / 主链路扫描成本
    
- space 看的是 memory footprint（内存占用体积）
    

---

# 三、latency 是不是延迟？

对。

但你最好这样记：

- latency = 延迟 / 响应耗时
    
- 在后端语境里，常常指一次请求从进来到返回结果花了多久
    

比如：

- O(n²) 的算法，在数据量大时，单次请求会更慢
    
- 这就会把接口的 latency 拉高
    

你可以把它理解成：

“用户发请求后，要等多久才能拿到结果。”

常见相关词一起记一下：

|   |   |   |
|---|---|---|
|术语|含义|一句中文解释|
|latency|延迟|单次请求的响应耗时|
|throughput|吞吐量|单位时间能处理多少请求|
|memory footprint|内存占用|代码/服务实际吃掉多少内存|
|bottleneck|瓶颈|系统最卡的那一段|

# 一、Q4 为什么“先查后存”很关键？

## 已知事实

这题要求：

- 选 两个不同下标
    
- 不能重复使用同一个元素
    

Hash Map 解法的标准动作是：

1. 遍历到当前元素 x
    
2. 先查 target - x 是否已经在 map 中
    
3. 如果有，返回“历史下标 + 当前下标”
    
4. 如果没有，再把当前元素存进 map
    

这叫：

先消费历史，再注册当前。

这就是比较工程化的表达。

## 为什么不能随便“先存后查”？

因为一旦你先把当前元素存进去，再去查补数，某些情况下会查到的就是当前元素自己。

### 例子

lyricsWeights = [5, 11, 13]

targetMood = 10

  

遍历到第一个 5 时：

如果你是 先存后查：

### 第一步：先存

map = {5: 0}

  

### 第二步：查补数

当前值是 5，补数也是：

10 - 5 = 5

  

这时你一查，发现 5 在 map 里，于是你可能会错误地得到：

[0, 0]

  

但这是非法答案，因为它复用了同一个下标。

---

## 本质原因

“先查后存”本质是在保证：

map 里只放“历史元素”，不放“当前元素自己”。

这样当你查到补数时，命中的一定是之前出现过的位置，而不是当前这个位置。

所以可以自然保证：

- 返回的是两个不同下标
    
- 不会把当前元素和自己配对
    

---

## 面试标准说法

你可以这样答：

“先查后存”的关键在于保证 Hash Map 中只包含当前元素之前出现过的历史元素。这样在查询补数时，如果命中，得到的一定是另一个下标，而不会把当前元素和自己配对。若改成“先存后查”，当 target = 2 * currentValue 时，可能会直接查到当前元素自己，导致错误使用同一 index 两次。

这段话你最好直接记住。

# 二、Q5 为什么有 negative number 也成立？

## 结论

成立，而且完全不影响解法成立性。

---

## 原因

这题 Hash Map 的核心不是“数字大小关系”，而是：

给当前值 x，查补数 target - x 是否存在。

这个逻辑对：

- 正数
    
- 负数
    
- 0
    
- 正负混合
    

都一样成立。

### 举个例子

lyricsWeights = [-3, 4, 7, 10]

targetMood = 4

  

当遍历到 7 时：

complement = 4 - 7 = -3

  

只要 -3 之前出现过，map 一样能查到。

说明：

- Hash Map 不关心正负
    
- 它只关心“这个 key 存不存在”
    

---

## 什么情况下才会受影响？

真正可能受影响的不是负数，而是这些问题：

### 1）overflow 风险

如果数值范围特别大，target - x 可能溢出。

### 2）Hash 冲突

理论上会有 hash collision，但通常平均复杂度分析里默认可接受。

### 3）题目语义变化

比如从“两数之和”变成“要求有序”“要求最近”“要求连续子数组”，那就不是这套了。

---

## 面试标准说法

你可以这样答：

即使数组中存在 negative number，Hash Map 解法依然成立。因为该方案的核心是对每个当前元素计算补数 target - currentValue，再判断这个补数是否已经出现过。这个查找过程只依赖值的精确匹配，不依赖数值是正数还是负数，因此负数不会破坏算法正确性。

# 二、Q7 为什么原题更适合 Hash Map，而不是排序后 Two Pointers

这是今天最重要的一题。

## 先给结论

不是说排序 + Two Pointers 不能做，  
而是 在原题语义下它不是优先方案，原因有 4 个：

1. 题目要求返回原始下标 index
    
2. 排序会破坏原始位置语义
    
3. 恢复原始下标需要额外映射，增加实现复杂度
    
4. 时间复杂度上，排序方案通常是 O(n log n)，不如 Hash Map 的 O(n)
    

---

## 1）原题要的是原始 index

Two Sum 原题不是要你返回：

- 两个值
    
- 或排序后的位置
    

而是要返回：

- 原数组中的两个下标
    

这意味着“位置语义”是题目 contract 的核心部分。

---

## 2）排序会破坏原始位置语义

一旦你先排序，元素顺序就变了。

比如：

原数组:

index 0 -> 11

index 1 -> 3

index 2 -> 7

target = 10

  

排序后可能变成：

3, 7, 11

  

你当然更容易用 Two Pointers 找到 3 + 7 = 10，  
但问题来了：

你现在拿到的是排序后的相对位置，不是原数组里的下标。

所以排序方案天然要补一个“值和原始下标的映射层”。

---

## 3）恢复原始下标会增加实现风险

为了不丢原始 index，你通常要把每个元素包装成：

(value, originalIndex)

  

然后排序这个结构体数组，再做双指针。

这就会带来额外问题：

- duplicate value 怎么处理
    
- 排序后返回的是包装结构，不是原数组
    
- 实现更绕，出 bug 概率更高
    
- 面试时讲解成本更大
    

大厂黑话可以这么说：

排序方案虽然能把查找问题转成双指针收敛问题，但它会破坏原始主键语义，需要额外维护位置映射，导致实现链路变长、认知负担变高。

---

## 4）复杂度也不占优

### Hash Map

- Time Complexity: O(n)
    
- Space Complexity: O(n)
    

### 排序 + Two Pointers

- 排序：O(n log n)
    
- 双指针扫描：O(n)
    
- 总体：O(n log n)
    

所以从性能上看，排序方案也不如 Hash Map。

# Q8 为什么会“被覆盖”？

## 已知事实

标准 Hash Map 的语义是：

key -> value

  

不是：

key -> 多个并列 value

  

所以如果你连续写：

5 -> 0

5 -> 3

  

那第二次通常会把第一次覆盖掉，最后只剩：

5 -> 3

  

这就是“覆盖”。

---

## 你刚才的理解哪里偏了

你说：

不应该在哈希表里面在后面再添加一个值一样的但是 index 不一样的键值对吗？

如果是标准 Hash Map，不行。  
因为 key 必须唯一。

也就是说：

- 可以有多个不同 key
    
- 但不能有两个完全相同的 key 同时并列存在
    

如果你真想保留同一个值对应多个位置，那数据结构就不能再是简单的：

value -> index

  

而要改成：

value -> list of indices

  

这已经是另一种设计了。

---

## 那为什么原题通常仍然没问题？

这是重点。

### 结论

因为原题只需要返回一组合法解，不需要返回所有解。

所以即使 duplicate value 导致旧下标被覆盖，通常也不影响找到一组正确答案。

---

## 举例说明

lyricsWeights = [5, 5, 11]

targetMood = 10

  

按“先查后存”走：

### 遍历到 index 0, value = 5

- 查补数 10 - 5 = 5
    
- map 里没有
    
- 存入：5 -> 0
    

### 遍历到 index 1, value = 5

- 查补数 10 - 5 = 5
    
- map 里有 5 -> 0
    
- 直接得到答案：[0, 1]
    

注意这里在找到答案之前，根本还没发生覆盖问题的影响。  
因为第二个 5 一来就已经把答案凑出来了。

---

## 再看另一个例子

lyricsWeights = [1, 5, 5, 9]

targetMood = 10

  

过程：

- 1 存进去
    
- 第一个 5 存进去：5 -> 1
    
- 第二个 5 来时，先查补数 5，命中 index 1
    
- 得到 [1, 2]
    

还是没问题。

---

## 面试标准说法

你可以这样答：

标准 Hash Map 中同一个 key 只能对应一个 value，因此当 duplicate value 再次写入时，旧的 index 可能被覆盖。  
但在 Two Sum 原题里，通常仍然没问题，因为题目只要求返回一组合法解，而且我们采用“先查后存”的策略。这样当第二个重复值出现时，往往会先用前一个 index 直接构成答案，在覆盖真正产生影响前就已经返回结果。  
只有在题目要求返回所有解时，才通常需要把结构升级为 value -> list of indices。

这段很关键。

# Q9 online processing 和 offline processing 是什么意思？

先给你最实用的理解。

## online processing

边读数据，边处理，边出结果。

意思是：

不需要等所有输入全部准备好，处理过程可以随着数据到来实时推进。

---

## offline processing

先拿到完整数据，再统一处理。

意思是：

必须先看见全量输入，才能开始做全局计算、排序、重排或批处理。

---

## 用这题来理解

### Hash Map 解法更像 online processing

因为它是这样工作的：

- 读到一个元素
    
- 立刻查补数
    
- 没有就存起来
    
- 继续读下一个
    
- 一旦匹配成功，马上返回
    

这说明它具备：

流式处理能力

也就是数据一边进，你一边算。

---

### 排序 + Two Pointers 更像 offline processing

因为排序前，你必须先拿到全部数据。

流程是：

1. 先收齐整个数组
    
2. 再排序
    
3. 再双指针扫描
    

这说明它依赖全量输入，是典型的 offline 风格。

---

## 面试里怎么答

你可以这样说：

Two Sum 的 Hash Map 解法更接近 online processing。因为它不需要等待全部数据完成预处理，而是在单次遍历过程中，一边读取当前元素，一边基于历史元素做补数匹配，命中后即可立即返回结果。  
相比之下，排序 + Two Pointers 更偏 offline processing，因为它依赖先获得完整输入并完成排序，再进行后续求解。

如果存在，说明之前某个位置已经出现过能和当前值配对的元素，这时就可以直接返回“历史下标 + 当前下标”。  
如果不存在，就把当前元素写入 Hash Map，也就是把 currentValue -> currentIndex 存进去，供后续元素查询。  
如果整个数组遍历结束后仍然没有命中，就进入 failure case，返回题目约定的失败结果，比如 null、空数组或者 [-1, -1]。

这段你要尽量说顺。

# 二、Q12 怎么回答得更像大厂面试

你原话：

哈希表需要空间来存储，同时比暴力循环解题快，所以说是空间换时间

方向没错，但太白。  
面试里建议你升级成下面这种说法。

---

## 标准回答

这题是典型的“用空间换时间”。  
如果不用额外空间，Brute Force 需要两层扫描去找配对元素，时间复杂度是 O(n²)。  
而引入 Hash Map 之后，我们相当于提前维护了一层 lookup index，把“查找某个补数是否存在”从一次线性扫描，降成均摊 O(1) 的哈希查询。  
代价是需要额外的 O(n) 空间去存储历史元素，所以本质上是拿 memory budget 换更低的 latency。

这段话你以后很多题都能复用。
