# AI 知识库（Dify RAG）

> 一句话：面向文档问答场景的 AI 知识库项目，重点关注文档处理、检索问答链路和回答边界控制。

---

## 简版（1 分钟）

这是一个面向文档问答场景的 AI 知识库项目，我主要关注文档接入、检索问答链路和回答稳定性。

项目链路：文档接入（PDF/扫描件 OCR + 文本清洗）→ chunk 切分 → embedding 向量化 → 建索引 → 查询时检索 TopK → rerank → 拼上下文给模型生成答案。

我比较关注 failure cases：召回不足、回答不稳定、乱码等。通过提示词约束、召回结果控制、异常兜底让系统在"不确定"时更保守可控。

这个项目体现的是：我会把 AI 项目当后端系统来看，关注链路、异常、边界和稳定性，不只停留在模型调用层面。

---

## 完整版（2-3 分钟）

这个项目目标是基于知识库的问答系统，让用户基于文档内容检索和提问。后端用 Java Spring Boot，RAG 能力集成 Dify 平台。

**文档接入与 ingestion：** 文档上传后异步处理入库。普通文档直接切分，PDF 扫描件走 OCR 分支提取文本。用 TransactionalEventListener(AFTER_COMMIT) 确保事务提交后再触发异步任务，@Async + ThreadPoolTaskExecutor 做异步执行。任务有状态机管理（PENDING → PROCESSING → COMPLETED/FAILED）。

**检索问答链路（RAG）：** 文档 → chunk 切分 → embedding 向量化 → 向量索引 → 查询时检索 TopK → rerank → 拼相关上下文给模型生成答案。返回结果附带 Sources（引用来源），让回答可溯源。

**边界控制与拒答：** 召回内容不足时不强答，通过提示词约束让模型回复"无法回答"。检索分数低于阈值时做兜底处理，避免一本正经胡说。

**工程化：** SSE 实时推送任务状态；外部平台（Dify）副作用控制——失败时需清理已上传资源；认证和知识库管理的 CRUD。

**trade-off：** RAG 能力依赖 Dify 而非自研，降低开发成本但形成外部依赖。权限模型较轻。没追求"最强效果"，优先保证链路完整、异常可控。

**诚实说不足：** 无大规模线上验证；Redis 未真正落业务；监控停留在基础层面；应用内异步扩展性有限，规模大了需要独立 Worker。

## 高频追问

**Q: RAG 核心链路怎么理解？**
A: 核心不是"接模型"，而是让模型回答建立在可检索知识上。文档 → chunk → embedding → 索引 → 检索 TopK → rerank → 拼上下文生成。价值是利用外部知识降低幻觉，但 chunk 切不好、召回不准时效果仍会受影响。

**Q: TopK、rerank、embedding 分别是什么？**
A: embedding 是把文本转成向量表示；TopK 是从向量库中检索最相似的 K 条；rerank 是对召回结果二次排序提升相关性。

**Q: RAG 为什么不能完全解决幻觉？**
A: 检索可能召回不相关内容，chunk 切分可能丢失上下文，模型仍可能基于不完整信息生成错误回答。RAG 降低但不消除幻觉。

**Q: 为什么要做 OCR？**
A: 扫描版 PDF 不是结构化文本，不做 OCR 无法参与检索。目的是让原始资料变成可被系统利用的数据。

**Q: 为什么不用 MQ / 独立 Worker？**
A: 当前任务量不大，应用内异步足够。引入 MQ 增加部署和维护成本，但规模大了需要考虑。

**Q: 为什么不自研 RAG？**
A: Dify 已封装 embedding + 向量检索 + rerank，自研需大量基础设施投入。当前目标是验证链路完整性和工程化能力，不是算法创新。

**Q: 为什么返回 Sources？**
A: 让回答可溯源可验证，用户能判断答案是否靠谱，也方便排查效果问题。

**Q: 回答效果不好怎么排查？**
A: 按链路逐段排查：文档预处理 → chunk 切分 → 检索策略 → 提示词 → 生成阶段，不直接归因给模型。

**Q: 失败任务为什么不都能重试？**
A: 有些失败涉及外部平台副作用（如 Dify 已部分上传），重试可能产生脏数据，需清理后才能重试。

**Q: 如果继续扩展怎么改？**
A: ①异步任务迁独立 Worker ②接入标准监控 ③考虑多知识库路由和权限细化。
