**

题目：Minimum Window Substring

给你两个字符串 s 和 t，请你返回 s 中包含 t 所有字符的最小子串。如果不存在这样的子串，返回空字符串 ""。

示例：

- s = "ADOBECODEBANC", t = "ABC"
    
- 输出："BANC"
    

再看一个：

- s = "a", t = "aa"
    
- 输出：""
    

---

## 结论

这题大概率属于：Sliding Window（滑动窗口） 题型候选，本质是“在字符串上维护一个可伸缩区间，找满足条件的最短窗口”。

---

## 已知事实

1. 题目本质  
    不是让你找“包含某几个不同字符”的子串，而是找一个区间，这个区间里必须覆盖 t 中的所有字符及其出现次数。
    
2. 核心思想候选  
    这题通常会用到：
    

- Hash Map：记录字符频次。  
    中文解释：用哈希表统计每个字符需要多少、当前窗口里有多少。
    
- Sliding Window：左右指针维护一个动态区间。  
    中文解释：右指针负责扩张窗口，左指针负责尽量缩短窗口。
    

4. 这题的难点  
    难点不在“找到一个满足条件的窗口”，而在：
    

- 找到后还能继续缩小
    
- 缩小过程中能准确判断“还是否满足要求”
    

6. 一个关键提醒  
    t = "AABC" 和 t = "ABC" 不是一回事。  
    前者要求窗口里至少有 2 个 A，所以必须处理重复字符计数。
    

---

## 合理推测

1. baseline 解法可以先枚举所有子串，再检查是否覆盖 t，但会很慢。
    
2. 更优解大概率是：
    

- 右指针不断扩张，直到“窗口刚好够用”
    
- 左指针再收缩，直到“再缩就不够用”
    

4. 真正的优化点在于：  
    不要每次都重新统计整个子串，而是随着窗口移动做增量更新。

- Sliding Window（滑动窗口）：在数组或字符串上，用左右边界维护一个连续区间，边移动边更新状态。

## 先往前推半步：baseline 和 optimal 的差别

### baseline solution（朴素解）

可以这样想：

- 枚举所有子串
    
- 对每个子串统计字符频次
    
- 判断是否覆盖 t
    
- 找最短的那个
    

问题：

- 子串很多
    
- 每个子串还要重新统计
    
- 重复工作非常多
    

### optimal solution（更优解）

优化点是：

- 不重复统计整个子串
    
- 窗口右移时，只更新新进来的字符
    
- 窗口左移时，只更新被移除的字符
    

这就是 incremental update（增量更新）。  
中文解释：窗口每次只改一点点，所以状态也只需要改一点点。

这里最容易错的是“什么时候改 valid”。

- 如果 c 不在 need 里，通常不用管合法性，只更新 window

说明这个字符第一次刚好达标，所以 valid++

注意不是 >= 就一直加。  
因为一个字符只能“第一次达标”那一刻加一次。

然后再执行：

你可以把它理解为：

## 已知事实

### 1. 题目本质

给定两个字符串：

- s：搜索空间
    
- t：目标需求
    

要求在 s 中找到一个最短连续子串，使得这个子串包含 t 中所有字符，并且每个字符出现次数 至少 满足 t 的要求。

注意：

- 不是只看字符种类
    
- 不是要求窗口内容和 t 完全相等
    
- 而是要求“覆盖需求”
    

例如：

- t = "AABC"
    
- 窗口里必须至少有：
    

- 2 个 A
    
- 1 个 B
    
- 1 个 C
    

---

### 2. 核心数据结构 / 思想

这题通常使用：

- Hash Map（哈希表）：记录字符频次  
    中文解释：统计“需要什么”和“当前窗口里有什么”。
    
- Sliding Window（滑动窗口）：左右指针维护一个连续区间  
    中文解释：右指针负责扩张，左指针负责收缩。
    
- valid：满足要求的字符种类数  
    中文解释：当前窗口里，有多少种字符已经达到了 need 的要求。
    

---

### 3. 三个核心状态

#### need

记录 t 中每个字符需要多少次。  
它是固定不变的。

例如：

- t = "ABC"
    
- need = {A:1, B:1, C:1}
    

#### window

记录当前窗口中每个字符出现多少次。  
它会随着窗口移动不断变化。

#### valid

表示当前有多少种字符已经“达标”。  
例如：

- need 有 A/B/C 三种字符
    
- 当前只有 A/B 达标
    
- 那么 valid = 2
    

窗口合法的判断条件是：

valid == need.size()

其中：

- valid：已达标的字符种类数
    
- need.size()：目标中不同字符的种类数
    

---

### 4. 最关键的主流程

整题是一个循环：

右扩到合法 → 左缩到极限 → 记录最短答案 → 再继续右扩

#### 右扩

右指针不断向右移动，把新字符加入窗口。

如果某个字符 c 在加入后满足：

window[c] == need[c]

说明这个字符种类第一次刚好达标，所以：

valid++

#### 左缩

当窗口合法时（valid == need.size()）：

1. 先更新当前最短答案
    
2. 再尝试移除左边字符
    
3. 如果移除的是一个“刚好够用”的必需字符，即移除前满足：
    

window[d] == need[d]

那么移除后会不达标，所以：

valid--

然后继续右扩。

## 合理推测

### 1. baseline solution（朴素解）

最自然的想法是：

- 枚举所有子串
    
- 对每个子串统计字符频次
    
- 判断是否覆盖 t
    
- 取长度最短的那个
    

这个方案可以做出来，但会很慢。

#### 为什么慢

慢在两个地方：

1. 子串数量本来就很多
    
2. 每个子串都要重新做频次统计，重复工作很多
    

本质瓶颈是：

大量重复计算

---

### 2. optimal solution（更优解）

优化思路是：

- 不重新扫描每个子串
    
- 用滑动窗口维护“当前区间”
    
- 每次窗口移动时，只更新一个进来的字符或一个出去的字符
    
- 用 valid 快速判断当前窗口是否合法
    

这属于 incremental update（增量更新）。  
中文解释：窗口每次只变化一点，所以状态也只局部更新，不重算整体。

---

### 3. 为什么 optimal 更快

因为每个字符最多被：

- 右指针访问一次
    
- 左指针移除一次
    

所以整体窗口移动次数是线性的，不会反复重算同一大段区间。

3. need 和 window 的职责
    

- need 固定不变
    
- window 动态变化

- 右扩时：刚好达标 → valid++
    
- 左缩时：删掉刚好够用的字符 → valid--
    

7. 左缩时的顺序
    

- 先更新答案
    
- 再删左边字符

这题是典型的滑动窗口题。  
我先用 need 统计 t 中每个字符需要的次数，用 window 统计当前窗口中的字符次数。

## 容易错的点

### 1. 把“至少满足”说成“完全一致”

不对。  
窗口可以有多余字符，只要覆盖住 t 即可。

### 2. 把 need 和 window 混了

不对。  
need 是固定需求；window 才是当前状态。

### 3. 用 >= 时反复增加 valid

不对。  
只有在“第一次刚好达标”时才能 valid++。

### 4. 左缩时先删再更新答案

不对。  
应该先记录当前合法窗口，再尝试删除左侧字符。

## 复杂度总结

设：

- n = s.length
    
- m = t.length
    

### baseline

如果暴力枚举所有子串并重新统计，时间复杂度通常会比较高，常见可到：

O(n^2 * 计算检查代价)

如果每次检查还要重新扫频次，整体会很慢。

### optimal

滑动窗口解法通常为：

Time Complexity：O(n)  
Space Complexity：O(k)

其中：

- n：字符串 s 的长度
    
- k：need / window 中涉及的字符种类数
    

如果字符集固定为 ASCII，也可近似看成常数空间。
