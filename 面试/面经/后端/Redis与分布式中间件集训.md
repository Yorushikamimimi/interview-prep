# Redis与分布式中间件速查

---

## 1. 高并发超卖问题分析

### 两种超卖
| 类型 | 定义 | 原因 |
|------|------|------|
| **真超卖** | DB库存真的为负数 | SQL设计漏洞、幂等失效、并发一致性漏洞 |
| **感知超卖** | 用户看到"成功"最后被取消 | 异步链路状态不一致，业务流程设计缺陷 |

### MySQL CAS扛不住10,000 QPS的原因
1. **CAS只做冲突检测，不做流量吸收**：失败请求也消耗DB资源
2. **热点行成为单点瓶颈**：所有请求竞争同一条库存记录
3. **失败重试放大流量**：用户重试 + 网关重试 + 服务重试 = Retry Storm
4. **线程池阻塞**：DB RT飙升 → Tomcat线程等待 → 线程池耗尽

### 应用层Root Cause
1. 同步直写DB，无异步缓冲/排队机制
2. 重试策略失控，1次意图放大成N次请求
3. 缺少幂等控制（userId + skuId + activityId）
4. 成功态返回过早（订单草稿创建就提示成功）

## 2. 30分钟止血方案

### 立刻做（关键止血）
1. **前端防重**：按钮置灰 + 防抖
2. **网关限流**：热点SKU限流，保护后端和DB
3. **关闭重试**：阻断Retry Storm
4. **服务降级**：秒杀接口改"排队中/结果稍后确认"
5. **快速熔断**：当DB RT持续恶化时拒绝部分请求

### 不是30分钟能上的（后续架构升级）
- Redis预热库存
- Lua原子扣减
- MQ削峰填谷
- 分布式锁
- 库存分段/分片

## 3. MySQL vs Redis方案对比

| 维度 | MySQL CAS | Redis + Lua |
|------|-----------|-------------|
| **吞吐量** | 低到中，DB直接承压 | 高，内存层完成判断和扣减 |
| **一致性** | 单库内局部强一致 | Redis、订单、DB天然不在一个原子事务 |
| **超卖风险** | 高热点下感知超卖明显 | 库存扣减层面风险低，全链路仍需补偿 |
| **用户体验** | 高并发下RT高、失败多 | 快速返回成功/失败/排队中 |
| **实现复杂度** | 低，逻辑集中在DB | 中到高，需Lua、MQ、幂等、补偿配套 |
| **运维复杂度** | 低 | 高，Redis高可用、持久化、热点key监控 |
| **故障恢复难度** | 低，数据主要在DB | 高，处理Redis预扣成功但DB未落单等问题 |
| **适用场景** | 中低并发普通业务 | 秒杀、抢券、短时高并发读写 |

## 4. 三种一致性

| 类型 | 定义 | 示例 |
|------|------|------|
| **数据一致性** | 不同存储里数据能对上 | Redis库存20，MySQL库存18 → 不一致 |
| **业务状态一致性** | 同一笔业务在不同服务状态能对上 | 库存已扣、订单创建失败、用户已支付 → 不一致 |
| **用户感知一致性** | 用户看到的结果自洽 | 先提示成功后说没库存 → 体验差 |

### 强一致性 vs 最终一致性
- **强一致性**：任何时刻数据都一致，代价是性能差（如Seata全局锁）
- **最终一致性**：允许短暂不一致，通过异步处理/重试/补偿最终收敛到正确状态

## 5. Redis + Lua原子扣减

### Lua能保证什么
- Redis内部多个命令原子执行（判断库存 + 扣减 + 写标记）
- 高并发下库存判断和扣减不会被插队

### Lua不能保证什么
- Redis扣成功 → MQ一定发送成功
- MQ发成功 → 订单服务一定消费成功
- 订单写库成功 → DB一定提交成功
- 服务崩溃时不会丢消息

### 典型链路
1. **Redis Lua扣减**：判断有库存 → 扣减 → 写用户占位标记
2. **发送MQ消息**：告诉订单服务"用户有资格下单"
3. **订单服务消费**：写MySQL订单
4. **失败补偿**：
   - 重试（设置最大次数）
   - 死信队列DLQ（人工处理）
   - 回补Redis库存

### 核心要点
**Redis + Lua管的是"资格"，不是"结果"**
- Redis：高并发抢资格
- MQ：削峰排队
- Order DB：真正落单
- Compensation：失败回滚
- Idempotency：防重复

## 6. Redis Key设计（秒杀场景）

| Key类型 | 示例 | 用途 | 过期时间 |
|---------|------|------|----------|
| **活动配置** | `seckill:activity:{activityId}` | 活动开始/结束时间、规则 | 活动结束后+1小时 |
| **商品详情** | `seckill:item:{skuId}` | 商品名称、图片、价格 | 活动结束后+1天 |
| **库存** | `seckill:stock:{skuId}` | 可售库存数量（热点写） | 活动结束后+1小时 |
| **用户幂等** | `seckill:user:{userId}:{skuId}` | 防止重复下单 | 活动结束后+1天 |
| **结果查询** | `seckill:result:{userId}:{skuId}` | 用户抢购结果 | 24小时 |
| **分布式锁** | `seckill:lock:{skuId}` | 补偿时加锁 | 30秒（加Watchdog） |

### 热点Key识别
- **读热点**：商品详情、活动配置（解法：多级缓存、本地缓存）
- **写热点**：库存扣减（解法：Lua原子操作、库存分段）

## 7. 缓存三大问题

### 缓存击穿（Cache Breakdown）
- **问题**：热点Key过期瞬间，高并发打穿DB
- **解法**：分布式锁（Mutex），只允许一个线程查DB并回写缓存

### 缓存穿透（Cache Penetration）
- **问题**：恶意请求不存在的Key（如ID=-999），Redis和DB都查不到
- **解法**：
  - 布隆过滤器（Bloom Filter）
  - 缓存空值（短TTL如5分钟）

### 缓存雪崩（Cache Avalanche）
- **问题**：大量Key同时过期（如都设置2小时TTL），DB瞬时压力暴增
- **解法**：TTL加随机抖动（如2h ± 10min）

## 8. 分布式锁与Redisson Watchdog

### 单Redis主从架构的不安全场景
1. Client A在Master获取锁
2. Master宕机，锁还未同步到Slave
3. Slave被提升为新Master
4. Client B在新Master获取到同一把锁
5. **两个客户端同时持有锁** → 一致性问题

### Redisson Watchdog机制
- **问题**：业务执行时间超过锁TTL，锁自动释放
- **Watchdog解法**：后台线程每10秒自动续期锁（默认30秒TTL）
- **风险**：进程假死（GC卡顿、网络分区）时Watchdog无法续期，但进程恢复后可能误以为仍持有锁

### Fencing Token（终极解法）
- 每次获取锁时返回递增Token
- 业务操作时携带Token
- 存储层拒绝旧Token的操作
- 适用场景：对一致性要求极高的核心业务

### 业务降风险手段
1. **幂等设计**：即使重复执行也安全
2. **CAS乐观锁**：DB层用version字段二次校验
3. **分段锁**：减少锁竞争（如库存分10段，每段独立锁）
4. **合理TTL**：锁时间略大于业务执行时间（如业务1秒，锁设3秒）

## 9. 热点Key治理

### 30分钟止血
1. **紧急限流**：对热点Key单独限流（如QPS降到安全值）
2. **读写分离**：读请求打到从节点
3. **本地缓存**：JVM Caffeine缓存热数据（秒级过期）

### 长期治理
1. **多级缓存架构**：
   - L1：本地缓存（Caffeine）
   - L2：Redis集群
   - L3：DB
2. **热点数据分片**：
   - 库存key后缀加随机数：`stock:{skuId}:1` ~ `stock:{skuId}:10`
   - 读时随机选一个分片，写时Lua批量扣减
3. **异步预热**：活动开始前提前加载热数据到Redis
4. **降级开关**：热点过载时降级到"排队中"

## 10. 秒杀系统完整链路

```
[用户] → [CDN静态资源]
         ↓
    [网关限流]（令牌桶/漏桶）
         ↓
    [Redis Lua扣库存]（幂等检查）
         ↓ 成功
    [发送MQ消息]
         ↓
    [订单服务消费]
         ↓
    [MySQL写订单]（本地事务）
         ↓ 失败
    [死信队列] → [补偿回滚Redis库存]
```

### 关键配置
- **网关限流**：热点SKU限流10,000 QPS
- **Redis库存Key TTL**：活动结束后+1小时
- **MQ重试**：最多3次，间隔递增（1s/5s/15s）
- **死信队列**：重试失败进DLQ，人工介入
- **幂等TTL**：24小时或活动结束后+1天

## 11. 高分面试答法模板

**秒杀超卖问题本质**：hotspot contention引发DB bottleneck，被retry storm放大的高并发库存事故。

**MySQL CAS局限**：只做冲突检测，不做流量吸收；热点行成为单点瓶颈；失败重试放大DB压力。

**Redis + Lua解法**：库存判断和扣减在内存层原子完成，大幅提升吞吐量。但只解决"抢资格"问题，全链路一致性需配合MQ、幂等、补偿、状态机设计。

**30分钟止血重点**：前端防重、网关限流、关闭重试、服务降级（改"排队中"），保护数据库优先于引入新组件。

**长期架构升级**：Redis预扣库存 + Lua原子操作 + MQ削峰 + 幂等控制 + 补偿机制 + 多级缓存 + 热点分片。

## 12. 生产避坑清单

### Redis相关
- ❌ Lua脚本不能直接发MQ（Redis不支持网络IO）
- ❌ 单Redis主从架构分布式锁不是绝对安全
- ❌ 所有Key都加相同过期时间会导致雪崩
- ✅ 用userId + skuId + activityId做幂等键
- ✅ Lua返回值设计：0=无库存 1=成功 2=已抢过 -1=活动未开始

### MQ相关
- ❌ Canal不该做主补偿方案（延迟高、不可靠）
- ✅ 消息带完整上下文（userId/skuId/activityId/timestamp）
- ✅ 死信队列处理永久失败消息
- ✅ 补偿前加分布式锁（防并发补偿）

### 一致性相关
- ❌ Redis扣成功≠最终成功，需全链路补偿
- ✅ 订单状态机：INIT → RESERVED → CREATED → PAID → CANCELLED
- ✅ 补偿逻辑：订单创建失败→回补Redis库存
- ✅ 用户提示保守（"处理中"优于"成功"）

### 性能相关
- ❌ 热点Key不做任何优化会打爆Redis单节点
- ✅ 库存分段（10段独立扣减）
- ✅ 本地缓存兜底（Caffeine缓存商品详情）
- ✅ 读写分离（读从节点，写主节点）
