# 门诊管理系统

> 一句话：面向诊所场景的门诊管理系统，重点解决患者管理、处方库存扣减和收费结算的数据一致性问题。

---

## 简版（1 分钟）

这个项目是一个门诊业务中后台系统，我主要负责后端主导的全栈开发，核心模块包括患者管理、医生开方、药房库存、收费结算和叫号大屏，目标是把患者建档、就诊、处方、扣库存、结算这条链路串起来。

技术上我主要做了三件事。第一，用 PostgreSQL JSONB 存患者动态档案和处方快照，解决动态字段多、频繁改表的问题，并配合 GIN 索引做检索优化。第二，在库存链路里用 UPSERT 做入库合并、用条件原子更新做扣减，避免并发下超卖。第三，在收费结算时加了 Redis 分布式锁和事务，防止重复结算、库存错扣。整体上这是一个偏单体、偏私有化部署的系统，我在这个项目里重点积累的是数据建模、一致性控制和业务 trade-off 的经验。

---

## 完整版（2-3 分钟）

这个项目是一个门诊场景的中后台系统，我主要负责后端主导的全栈实现，核心模块包括患者管理、医生开方、药房库存、收费结算和叫号大屏，目标是把患者建档、接诊、处方、扣库存、收费这些原本分散的流程串成一个闭环。

我在这个项目里重点做了四件事：

**数据建模：** 患者档案和处方明细有很多动态字段（过敏史、标签、扩展信息），全部拆表开发成本高。所以用了 PostgreSQL JSONB 存半结构化数据，配合 GIN 索引和 jsonb_exists 做检索，兼顾扩展性和查询性能。核心字段（ID、状态、编码）仍保留关系型设计。

**库存一致性：** 入库用 UPSERT（INSERT ... ON CONFLICT DO UPDATE），按药品编码冲突时直接合并库存。出库扣减用条件原子更新（current_stock >= quantity），一次 SQL 完成判断和扣减，避免"先查后改"的并发超卖问题。

**结算防重：** 收费环节加了 Redis 分布式锁（按账单号），拿到锁后在事务里完成：读取处方 → 逐条扣减库存 → 更新就诊单状态。任一步失败整体回滚，防止"钱扣了库存没扣"。

**实时叫号：** Redis ZSet 维护等待队列（时间戳做 score），WebSocket 广播呼叫结果到大屏。

**trade-off：** 不是大而全的医院 HIS，聚焦门诊最短闭环。单体架构 + 数据库事务 + Redis 锁，在当前规模下优先保证简单、稳定、好维护。

**诚实说不足：** 基于 RuoYi 二开，基础能力有框架复用；测试体系不完整；更适合中小规模私有化场景。

## 高频追问

**Q: 为什么用 JSONB？**
A: 患者标签、过敏史、处方扩展字段动态变化频繁，全拆表联表成本高。只把动态部分放 JSONB，核心字段保留关系型。

**Q: JSONB 缺点？**
A: 约束能力弱于普通列，查询写法复杂，要控制使用边界，不能滥用。

**Q: 为什么用 GIN 索引？**
A: JSONB 无索引会退化全表扫描。GIN 适合包含关系和键值查询场景。

**Q: 为什么库存不用"先查再改"？**
A: 有竞争窗口，并发下易超卖。条件原子更新把判断和扣减合成一个 SQL。

**Q: 入库为什么用 UPSERT？**
A: 同一药品重复补货时避免分支判断和并发冲突，直接用数据库冲突机制合并库存。

**Q: 为什么支付要 Redis 锁？库存不是已原子扣减了吗？**
A: 库存原子扣减解决单药品层面正确性，Redis 锁解决整笔账单重复处理问题（如重复点击），两者解决不同问题。

**Q: 为什么不上 MQ / 分布式事务？**
A: 单体门诊系统，当前规模下 DB 事务 + Redis 锁已够用，复杂方案收益不高。

**Q: 为什么不用乐观锁版本号？**
A: 条件更新更直接，语义清晰，天然避免"先读后写"窗口。

**Q: 事务里做了哪几步？**
A: 读取待结算处方 → 逐项扣减库存 → 更新就诊单状态，任一步失败回滚。

**Q: 如果再做一版优先改什么？**
A: 补自动化测试、完善审计日志、根据规模决定是否异步化部分流程。
